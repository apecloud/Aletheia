# Finding Approval Loop PRD - task #149

## Product Position

Findings should be reviewable and approvable, but a Finding approval is not the
same thing as an Ontology artifact approval.

- Ontology approval means a schema, object, link, property, or rule can become
  canonical and may be used as a graph ingestion or governance basis.
- Finding approval means a reasoning conclusion has been reviewed and can be
  used as a citeable, audited insight in downstream reasoning, dashboards, and
  case workflows.

The core boundary is:

```text
Finding approve = reviewed insight / decision memory
Finding approve != canonical ontology write
Finding approve != canonical graph write
Finding approve != automatic business execution
```

If an approved Finding implies that ontology, graph, or rule data should change,
the product must create a separate change proposal and send it through the
stronger canonical governance gate.

## User Value

Autopilot and Reasoning should not stop at "the agent generated a draft". The
closed loop is:

1. Agent discovers candidate Findings.
2. Human reviewer approves, rejects, or asks for more evidence.
3. Approved Findings enter a registry as reusable reviewed insights.
4. The system uses approved Findings to drive next actions, future reasoning,
   dashboard attention, and change proposals.
5. New evidence or rule/data changes can reaffirm, stale, or supersede old
   Findings.

This turns the Reasoning page from a generation surface into a decision workflow.

## Lifecycle

Required Finding states:

- `draft`: candidate Finding generated by Reasoning or Autopilot.
- `needs_more_evidence`: reviewer accepts the direction but requires more proof.
- `rejected`: reviewer rejects the conclusion or value.
- `approved`: reviewer confirms the Finding is currently usable as a reviewed
  insight.
- `stale`: source data, evidence, ontology, or rule basis changed enough that the
  Finding needs revalidation.
- `superseded`: a newer Finding replaces this one.

Required review event:

- `reaffirmed`: reviewer or revalidation confirms an existing approved Finding
  still holds under the current data, ontology, and rule basis. This can keep the
  status as `approved` while appending a new review/history event with the fresh
  basis metadata.

Allowed transitions:

```text
draft -> approved
draft -> needs_more_evidence
draft -> rejected
needs_more_evidence -> draft
needs_more_evidence -> approved
approved -> stale
approved -> superseded
approved -> rejected  # only through explicit reviewer reversal
stale -> approved     # after revalidation
stale -> superseded
approved -> approved  # reaffirmed event with refreshed basis metadata
```

Every transition must append a review event with reviewer, timestamp, reason,
previous status, next status, and evidence snapshot.

## Approval Requirements

The Approve action must be disabled or rejected unless the Finding has:

- a clear conclusion and user-facing title;
- tenant and scope;
- source reasoning task/run or Autopilot session/hypothesis;
- at least one evidence chain item;
- evidence source labels, including whether each item is raw fact, approved
  ontology basis, prior Finding, inference, assumption, or counter-evidence;
- confidence/value score;
- evidence freshness metadata;
- explicit canonical boundary text: approving this Finding does not modify
  ontology or graph.

For creditcardfraud, evidence must use masked/safe data sources. Raw sensitive
fields such as `cardCVV` and `enteredCVV` must not appear in API payloads or DOM.

## Post-Approval Use

An approved Finding can be used in five places.

1. Finding Registry

Approved Findings become searchable, filterable, and auditable. The registry is
the user's reviewed insight memory, not a raw fact table.

2. Reasoning Context

Future reasoning may cite approved Findings as prior reviewed conclusions. The
UI and API must label them as `prior_finding` or `reviewed_inference`, not as raw
source facts. Answers that depend on them should show a dependency line such as:

```text
Uses approved prior Finding: <finding_key>, approved at <time>, based on <evidence snapshot>.
```

3. Workspace / Case Inbox

Approved Findings can create next actions:

- open investigation case;
- request more evidence;
- assign manual review;
- rerun Autopilot on a narrower scope;
- create rule tuning proposal;
- create ontology/graph change proposal;
- mark as monitored risk.

Actions have their own owner, status, due date, and result. Finishing an action
does not mutate the Finding silently; it appends an outcome event.

4. Home / Dashboard

High-value approved Findings should appear as key findings, attention items, and
recent changes. Dashboard cards must expose status, confidence, evidence count,
and next action state.

5. Change Proposal Bridge

When a Finding implies a structural change, the product should offer:

- "Propose ontology change"
- "Propose graph correction"
- "Propose rule update"

These create separate proposed artifacts or rule proposals. The approved Finding
is only the rationale and evidence package.

## UI Requirements

Reasoning page:

- Candidate Findings show review actions: Approve, Needs more evidence, Reject.
- Approve opens a review modal requiring an approval reason and evidence check.
- Approved Finding detail shows "Approved insight" and the canonical write
  boundary.
- If a Finding uses a prior approved Finding, show the dependency explicitly.

Autopilot page:

- Finding Inbox keeps candidates draft-only until human approval.
- Approve is a human review action, not an Autopilot API promote action.
- `auto_approve_findings=false` remains visible in safety profile.
- Candidate cards show whether they will create action, registry entry, or
  change proposal after approval.

Findings page:

- Add filters: Draft, Needs evidence, Approved, Stale, Superseded, Rejected.
- Approved detail shows: evidence chain, review history, used-by, linked actions,
  stale/superseded status, and rerun/revalidate controls.

Workspace:

- Approved Findings may appear as Case Inbox items only when they have an
  unresolved next action.
- Workspace should not become a duplicate Finding Registry.

## Staleness And Supersession

Approved Findings must not become permanent truth without revalidation.

Mark a Finding `stale` when:

- source snapshot changed materially;
- ontology basis changed;
- rule version changed;
- evidence chain contains data that is no longer available or now degraded;
- a reviewer manually requests revalidation.

Mark a Finding `superseded` when:

- a new approved Finding resolves the same question with stronger evidence;
- a new Finding contradicts and replaces the old conclusion;
- a rule or ontology change makes the previous conclusion obsolete.

Stale and superseded Findings remain visible for audit but should not be used as
active reasoning context unless the user explicitly includes them.

Append a `reaffirmed` event when:

- the source, ontology, or rule basis changed, but the conclusion still holds;
- a reviewer reruns validation and confirms the evidence is still sufficient;
- a downstream action outcome supports the same conclusion.

Reaffirmed Findings remain active approved context, but the UI must show the
latest reaffirmed time and basis so users know the conclusion is not relying only
on an old approval.

## API Requirements For Implementation

Recommended minimum API:

```text
GET  /api/findings?tenant=<id>&status=approved
GET  /api/findings/<finding_key>?tenant=<id>
POST /api/findings/<finding_key>/review?tenant=<id>
POST /api/findings/<finding_key>/actions?tenant=<id>
POST /api/findings/<finding_key>/change-proposals?tenant=<id>
POST /api/findings/<finding_key>/revalidate?tenant=<id>
```

`POST /review` body:

```json
{
  "decision": "approve | needs_more_evidence | reject | mark_stale | supersede | reaffirm",
  "reason": "...",
  "reviewer": "...",
  "superseded_by": null,
  "requested_evidence": []
}
```

Rules:

- `approve` must never call canonical ontology or graph write paths.
- `approve` must preserve source task/run/session provenance.
- `approve` must preserve evidence snapshot/version metadata.
- `needs_more_evidence` can enqueue a follow-up hypothesis for Autopilot.
- `change-proposals` can create proposed ontology/graph/rule artifacts, but those
  start in proposed state and require their own approval gate.
- active reasoning context queries must default to active `approved` Findings
  plus `reaffirmed` approval events only, and must exclude `rejected`,
  `needs_more_evidence`, `stale`, and `superseded` Findings unless the caller
  explicitly requests audit/history mode.

## Acceptance Criteria

Implementation task #150 is complete when:

- a draft Reasoning Finding and an Autopilot candidate can be approved by a human
  review action;
- approved Findings appear in an Approved Findings / Finding Registry view;
- approved Findings can be cited by a later reasoning task with explicit
  `prior_finding` labeling;
- approved Findings can create at least one Workspace next action;
- an approved Finding can create an ontology/graph/rule proposal without writing
  canonical state;
- review history, evidence chain, and source run/session are visible;
- stale/superseded states and reaffirmed review events are represented in API and
  UI;
- creditcardfraud payload and DOM do not expose raw sensitive fields.

Validation task #151 is complete when:

- approving a Finding does not change canonical ontology artifacts;
- approving a Finding does not write canonical graph data;
- canonical ontology/graph tables or APIs show no new canonical writes, graph
  ingestion, artifact approval, or artifact version change before vs. after
  approving a Finding;
- approved Findings are reusable in reasoning context only as reviewed
  inferences;
- rejected and needs-more-evidence Findings are not shown as approved context;
- stale/superseded Findings are excluded from default active context;
- reaffirmed Findings remain active but show their latest reaffirmed basis;
- default active context returns only active approved/reaffirmed Findings; all
  other statuses require explicit audit/history query mode;
- evidence and review audit are preserved after approval;
- masked/safe-source boundaries remain intact for creditcardfraud.
